# ThreatLens Behavior Contracts

Created from the current codebase plus the locked requirements captured in `/tmp/threatlens_refactor_handoff.md`.

## Purpose

This document freezes the expected behavior of the current `scan`, `breach`, and `shield` product areas before structure changes.

Unless the user explicitly changes product behavior, refactors must preserve these contracts.

## Scan

### Inputs

- Manual input from `app/(tabs)/scanner.tsx`
- Foreground Android notification interception from `src/modules/notificationBridge.ts`
- Background Android interception from `src/tasks/notificationTask.ts`
- Swiped-away / killed-state Android recovery through native pending scans generated by `withThreatLensConfig.js` and consumed in `app/_layout.tsx`
- Shared-text / paste prompt deep links routed through `app/_layout.tsx`

### Classification

- Messages are classified as exactly one of:
  - `SAFE`
  - `PROMO`
  - `SPAM`
  - `SCAM`
  - `PHISHING`
  - `UNAVAILABLE`
- `src/services/nimService.ts` remains the source of truth for classifier behavior and guardrails.
- `SAFE` and `PROMO` results must carry no red flags and no suggested actions.
- `UNAVAILABLE` means AI analysis degraded or could not run; it must not be silently converted into any threat class.

### Notification Interception Rules

- Foreground active app:
  - JS foreground listener handles interception.
- Background alive:
  - Headless JS path handles interception.
- Swiped away / process dead:
  - Native fallback may classify and queue results for later recovery.
- Filtering, low-signal suppression, and dedup rules must continue preventing noisy/system notifications from being classified.
- Stacked messaging notifications must prefer the latest unread message, not the combined unread stack.

### History And Persistence

- Scan history is a single ordered history across all scan sources.
- The history must include:
  - manual scans
  - foreground auto-intercepted scans
  - background auto-intercepted scans
  - killed-state recovered scans
- `SAFE` auto-intercepted scans must also be stored silently in history.
- History ordering must reflect scan execution order, newest first in the UI.
- History must survive relaunch.
- Duplicate scan records must be suppressed by stable scan id.

### Alerts And Routing

- `PROMO` may trigger a promotional alert.
- `SPAM`, `SCAM`, and `PHISHING` may trigger a threat alert.
- Tapping a scan alert opens the scan result screen with the encoded result payload.
- If the app cold-starts from a notification-opened scan result, Android back and header back must land on Scanner, not exit the app.
- Paste-prompt alerts must route to Scanner with prefilled text.

### Dashboard Coupling

- Non-`UNAVAILABLE` scan results contribute to dashboard scanned-message state.
- Suggested actions from scan results register as dashboard suggestions.
- Scan suggestion completion affects score the same way it does today.

## Breach

### Inputs

- User-managed monitored credentials from `app/(tabs)/breach.tsx`
- Remote provider fetches from `src/services/breachApiService.ts`
- Background breach checks from `src/services/backgroundTasks.ts`

### Credential Rules

- Credentials can be email or username.
- Duplicate credentials are prevented case-insensitively.
- Adding a credential triggers a breach scan.
- Removing a credential must immediately remove related breach cards and related breach guidance from the visible app state.

### Breach Scan Outcome

- Results merge newly fetched breach items with previously persisted breach state.
- Persisted fields that must survive refetch include:
  - `resolved`
  - `aiGuidance`
- Breaches are sorted newest first by breach date.
- Active breach count for the dashboard reflects unresolved breaches.

### Notifications

- When `notifyOnNew` is enabled and new breaches are found, the app sends a local breach alert.
- Tapping a single-breach alert deep-links to that breach detail.
- Tapping a multi-breach alert deep-links to the breach list.

### Persistence

- Credentials persist across relaunch.
- Cached breaches persist across relaunch.
- Hydration restores credential state and breach cache before UI use.

### Guidance

- AI guidance is fetched lazily per breach.
- Previously stored guidance must be reused.
- Fallback guidance stays marked as fallback and does not count as actionable progress.

### Dashboard Coupling

- Dashboard breach score math and pending-action behavior must remain unchanged.
- Removing credentials must prune related breach suggestions from the dashboard.

## Shield

### Scope

The Shield area remains exactly:

- sign image
- verify signed image
- trust/settings management

### Protect Flow

- User selects an image from the media library.
- App signs the image via `src/services/imageTrustService.ts`.
- A signed payload is embedded in the output image.
- Successful protect flow increments the dashboard protected-image count.
- User can save the signed image.

### Verify Flow

- User selects an image.
- App verifies local cryptographic integrity.
- Optional cloud revocation check remains user-toggleable.
- Verification states remain:
  - `AUTHENTIC`
  - `TAMPERED`
  - `INVALID_SIGNATURE`
  - `CLONE_APP`
  - `REVOKED`
  - `OFFLINE`
  - `NO_PROTECTION`
  - `CORRUPT`

### Settings Flow

- Registry base URL, registry API key, and master public key remain editable.
- Settings load from secure storage and save back to secure storage.
- Device trust snapshot remains visible when available.

### Android Protected Folder Behavior

- Android protected-folder selection behavior remains unchanged.
- Existing folder selection can be changed or reset.
- Save-to-folder retry behavior after expired permissions remains unchanged.

## Refactor Guardrails

- Do not change classification semantics in `src/services/nimService.ts`.
- Do not change score math in `src/utils/scoreCalculator.ts`.
- Do not change breach-removal behavior.
- Do not expand or shrink Shield product scope.
- Do not destabilize the working notification execution modes.
- Prefer extracting deep modules behind explicit seams over adding more thin pass-through helpers.
